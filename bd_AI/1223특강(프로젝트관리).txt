프로젝트 = 조직 내 목표 혹은 고객과의 계약상의 목표 달성을 위해
한정된 기간과 비용 내에서 장해진 자원을 활용하여 완수하고자 하는 과제

사업기획 및 발주 -> 제안 -> 계약 및 협상 -> 프로젝트수행

제안평가
* 평가주체 - 사업을 발주한 고객 / 외부 전문가 등 평가위원(공공기관)
* 평가대상 - 기술평가 : 제시된 실행안(기술)->제안서, 요약서, pt 자료
              - 가격평가 : 제안 가격->제안가
* 평가요소 - 기술의 적합성, 차별성, 가격의 적합성

사업참여 의사 결정
- 현실적 수주확도
- 성공적 수행 가능?
- 매출 규모와 이익률
- 고객과의 관계유지를 위한 전략적 필요

IT 프로젝트 
- 컨설팅  : 컨설턴트=백조, 
- 시스템 구축(SI) : 신규 시스템 대상 혹은 기구축 고도화, 업무강도 높음, 경험 多 
- 운영 및 유지보수(SM) : 기구축 기능 개선 및 유지보수, 업무강도 낮음, 경험 少

프로젝트 규모 小 -> 성공확률 高
                  大                 低

IT 프로젝트 특징
- 진행 상황을 가시적으로 정확히 판단하기 어려움
- 투입 인력(개발자) 의존도 高
- 초기에 사용자 요구사항을 명확히 정의하기 어려움
- 일괄계약 형태로 진행하여 초기 정확한 규모 산정 어려움

===> 프로젝트 관리(Project Management)
프로젝트 관리 영역
목표 - 범위 / 일정 / 원가 / 품질
수단 - 자원 / 의사소통 / 위험 / 조달 / 이해관계자

프로젝트 수행 방법론
*waterfall 모델
요구분석 -> 설계 ->개발 ->테스트
*agile 모델
-일정한 반복주기를 가지고 필요할 때마다 수정 및 보안
- 고객이 불리하고 번거로워함 -> 회사 내부에서만 사용
- 업무방식의 변화 필요
- 보고체계의 단순화 필요

프로젝트 착수 단계
1) 프로젝트 리더 선임, 팀원별 역할&책임을 정함
2) 팀 협의하에 프로젝트 ground rule 정함  
(업무 후 일일보고서 작성, 다음날 함께 의논, 결과 공유)

프로젝트 계획 단계
1)요구사항(SPEC)을 분석하여 명확히
-> 요구사항 정의서(명세서) - 시스템 기능/비기능 요구사항을 이해관계자와 협의하여 확정 및 구체화
-> use case 명세서 - 요구사항에 댛 엑터와 시스템 활동 및 상호간 활동에 대해 순서화된 시나리오를 기술, 연관관계를 도식화한 다이어그램 작성
-> 요구사항 추적표 - 단계별 산출물이 일관성을 가지는지를 추적/관리, 요구사항이 추가/변경될 경우 영향력 평가에도 사용
2) WBS 기반으로 체계적인 공정계획 수립
*일정계획 수립 시 고려사항
- 개발 내역에 대한 기술적 난이도를 사전에 평가
- 업무 담당 팀원 각각의 능력 고려
- 예상치 못한 이슈가 발생하는 경우가 많으므로, 이슈 해결 버퍼를 고려

프로젝트 실행 단계
1) 개발 전 목표 시스템에 대한 체계적인 설계 수행
- 불필요한 코드를 줄이고 테스트/재작업이 쉬운 코드 생성
- 개발과정에서 무한 수정/변경 등의 삽질 방지
- 향후 수정 및 유지보수가 용이하여 운영 생산성 향상
->클래스 모델(클래스 설계서)
-> 사용자 인터페이스 설계서
-> 인터페이스 설계서
-> 아키텍처 설계서
-> 엔티티 관계 모형 설계서
-> 데이터베이스 설계서
2) 체계적인 변경 통제와 형상 관리 필수  by git
- 버전/이슈/빌드/릴리즈/배포 관리
